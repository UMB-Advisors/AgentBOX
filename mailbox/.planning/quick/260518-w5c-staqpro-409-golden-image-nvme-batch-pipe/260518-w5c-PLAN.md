<objective>
STAQPRO-409 — Build the golden-image NVMe batch pipeline so a single seed Jetson + NVMe can be captured to an image and cloned per-customer at packing time (~15 min/NVMe), avoiding the per-customer SDK-Manager flash bottleneck.

**Purpose:** The 3-5 customers / 90 days volume target makes the SDK-Manager-per-unit flow unworkable. This pipeline collapses per-customer provisioning to `dd` + a personalization script, with safety guards that prevent the operator from accidentally wiping the workstation.

**Output:** Two shell scripts + one runbook on branch `feat/staqpro-409-golden-image`. Scripts are testable in isolation (root checks, slug regex, dry-run output, root-device refusal); full hardware-loop verification (real NVMe in the workstation NVMe slot) is Dustin-at-the-bench work and is NOT a blocker for landing this plan.
</objective>

<safety_pattern>
Required safety gate structure (both scripts):

1. `set -euo pipefail` at top
2. Root check: `[[ $EUID -ne 0 ]] && { echo "ERROR: must run as root"; exit 1; }`
3. Arg parsing including `--dry-run` and `--yes`
4. Validation phase: device exists as block device, slug matches regex, paths sane
5. **Root-device refusal**: for `factory-image.sh`, compare source device's parent against `findmnt -no SOURCE /` — refuse if they match. For `factory-prep-nvme.sh`, refuse if any partition of the target device is mounted as `/` on the workstation. Use `lsblk -no PKNAME <device>` and `findmnt` to derive the parent disk reliably.
6. Print intended action ("WILL READ from /dev/X size=NNN to /path/out.img" / "WILL WRITE customer-slug=foo to /dev/Y mounted at /mnt/...")
7. `if [[ $DRY_RUN -eq 1 ]]; then exit 0; fi`
8. `if [[ $YES -ne 1 ]]; then read -r -p "Type 'YES' to proceed: " confirm; [[ "$confirm" == "YES" ]] || exit 3; fi`
9. Destructive operation with `trap … EXIT` for cleanup (umount on failure)
10. Final verification: image sha256 / `lsblk` of personalized device
</safety_pattern>

  <name>Task 1: Create scripts/factory-image.sh — workstation-side seed-NVMe → golden-image capture</name>
  <files>scripts/factory-image.sh</files>
  <action>
Create `scripts/factory-image.sh` as an executable bash script (`chmod +x`) following the header convention from `scripts/provision-customer-dns.sh` (the cleanest reference per CLAUDE.md §5).

**Header block** (PURPOSE / USAGE / EXAMPLES / EXIT CODES — match provision-customer-dns.sh structure):
- PURPOSE: Capture a seed NVMe (already factory-bootstrapped per STAQPRO-410's `factory-bootstrap.sh`) to a compressed golden-master image on the workstation. Runs ONCE per JetPack version on Dustin's workstation. The captured image is later `dd`'d onto blank NVMes at packing time and personalized via `factory-prep-nvme.sh`.
- Reference STAQPRO-409.

**CLI shape:**
```
./scripts/factory-image.sh [--dry-run] [--yes] [--no-compress] [--output-dir <dir>] \
    --source <device-or-by-id> [--jetpack-version <ver>]
```
- `--source` accepts either `/dev/nvmeXn1` or `/dev/disk/by-id/<id>` (prefer by-id — note in usage text).
- `--jetpack-version` defaults to reading from `/etc/nv_tegra_release` if running on a Jetson, otherwise required; used only for the output filename.
- `--output-dir` defaults to `/var/lib/mailbox-images` (must exist or script creates it with `install -d -m 0755`).
- `--no-compress` skips the zstd step (still produces .img + .sha256).

**Filename convention** (per ticket):
`mailbox-golden-v<JetPack-version>-<YYYY-MM-DD>.img` and `.img.zst` (if compressed).

**Required behavior:**
1. `set -euo pipefail`.
2. Root check — exit 1 if not root.
3. Parse args, validate exactly one `--source` given, validate target exists with `[[ -b "$SOURCE" ]]`.
4. **Root-device refusal (safety gate)**: compute the parent disk of the workstation's root (`ROOT_DISK="/dev/$(lsblk -no PKNAME "$(findmnt -no SOURCE /)" | head -1)"`). Resolve `--source` via `readlink -f` if it's a `/dev/disk/by-id/...` symlink. If the resolved source parent matches `ROOT_DISK` — `echo "REFUSED: --source resolves to the workstation's root device $ROOT_DISK"; exit 3`.
5. Print planned action: source path, resolved device, source size (`blockdev --getsize64`), output path, free-space at destination (`df -B1 --output=avail "$OUTPUT_DIR" | tail -1`). REFUSE if free space < 1.1 × source size — `exit 3`.
6. If `--dry-run`, exit 0 here.
7. Confirmation prompt unless `--yes` (require literal `YES`).
8. Capture phase:
   - If `command -v pv >/dev/null`: `pv -tpreb "$SOURCE" | dd of="$OUT_IMG" bs=64M conv=fsync iflag=fullblock`
   - Else: `dd if="$SOURCE" of="$OUT_IMG" bs=64M status=progress conv=fsync iflag=fullblock`
9. Compute sha256: `sha256sum "$OUT_IMG" > "$OUT_IMG.sha256"`.
10. Compression (unless `--no-compress`): `zstd -19 --rm "$OUT_IMG" -o "$OUT_IMG.zst"` then `sha256sum "$OUT_IMG.zst" > "$OUT_IMG.zst.sha256"`. If `zstd` is not installed, warn but do not fail (image is still usable).
11. Print summary block: image path(s), sha256, size, capture duration. Suggest next step: `dd if=<image> of=<blank-nvme> bs=64M status=progress` then `scripts/factory-prep-nvme.sh`.
12. Use a `trap 'rm -f "$OUT_IMG.partial"' EXIT` pattern — capture into `$OUT_IMG.partial` and `mv` to `$OUT_IMG` on success only, so a Ctrl-C or kernel-OOM mid-capture doesn't leave a misleading "complete" file behind.

**Exit codes:**
- 0 success
- 1 bad invocation / missing root
- 2 validation failure (source missing, jetpack version unknown, output dir creation failed)
- 3 safety gate tripped (root-device match, insufficient free space, user declined confirm)

`chmod +x` the file when done.

  <name>Task 2: Create scripts/factory-prep-nvme.sh — per-customer NVMe personalization after dd-restore</name>
  <files>scripts/factory-prep-nvme.sh</files>
  <action>
Create `scripts/factory-prep-nvme.sh` as an executable bash script (`chmod +x`). Same header convention as Task 1.

**PURPOSE block:** Personalize a freshly-`dd`'d NVMe (loaded with a golden image from `factory-image.sh`) for a specific customer before packing. Runs on the workstation — the NVMe is in a USB-NVMe adapter / spare M.2 slot. Mounts the NVMe's rootfs, regenerates per-host identity (SSH host keys, Tailscale state, hostname, /etc/hosts, journals, bash history), and stamps the customer slug into `/etc/mailbox-customer` for the appliance's `.env` loader to read at boot.

**CLI shape:**
```
./scripts/factory-prep-nvme.sh [--dry-run] [--yes] [--mount-point <path>] \
    --slug <customer-slug> (--device <path> | --by-id <path>)
```

**Required behavior:**
1. `set -euo pipefail`.
2. Root check.
3. Arg parsing — require `--slug` plus exactly one of `--device` / `--by-id`. Prefer `--by-id` for the example in usage text ("safer — names won't shift between reboots").
4. **Slug validation**: `[[ "$SLUG" =~ ^[a-z0-9][a-z0-9-]{0,30}[a-z0-9]$ ]]` — exit 2 with a clear message if it fails.
5. Resolve `--by-id` via `readlink -f` to a `/dev/...` path. Validate `[[ -b "$DEVICE" ]]`.
6. **Root-device refusal (safety gate)**:
   - Workstation root parent disk: `ROOT_DISK="/dev/$(lsblk -no PKNAME "$(findmnt -no SOURCE /)" | head -1)"`.
   - Target parent disk: `TARGET_DISK="$DEVICE"` (already a whole-disk path expected); if user passed a partition, strip via `lsblk -no PKNAME`.
   - If `TARGET_DISK == ROOT_DISK`, REFUSED exit 3.
   - Also refuse if ANY partition of `TARGET_DISK` appears in `findmnt -no TARGET` as `/` or `/boot` on the running workstation. Use `lsblk -lnpo NAME "$TARGET_DISK" | xargs -n1 findmnt -no TARGET 2>/dev/null | grep -E '^/$|^/boot'` — non-empty match = refuse.
7. **Identify rootfs partition**: most JetPack-on-NVMe layouts make `p1` the rootfs. Detect by mounting the largest ext4 partition: `lsblk -lnpo NAME,FSTYPE,SIZE "$TARGET_DISK" | awk '$2=="ext4"' | sort -k3 -h | tail -1 | awk '{print $1}'`. If empty, exit 2 — "no ext4 partition found; was this NVMe `dd`'d from a valid golden image?".
8. Print intended action: device, rootfs partition, mount-point, slug, files-to-be-modified list. If `--dry-run`, exit 0.
9. Confirmation prompt unless `--yes` (literal `YES`).
10. **Mount + trap**:
    - `MNT="${MOUNT_POINT:-$(mktemp -d -t mailbox-nvme-XXXXXX)}"`
    - `mount "$ROOTFS_PART" "$MNT"`
    - `trap 'sync; umount "$MNT" 2>/dev/null || true; [[ -z "${MOUNT_POINT:-}" ]] && rmdir "$MNT" 2>/dev/null || true' EXIT` — so a mid-script failure still unmounts cleanly (per the lesson from `feedback_probe_cleanup_trap` in user memory).
11. **Personalization operations** (each as a bash function, each logging "[personalize] <action>"):
    - **Hostname**: `echo "$SLUG" > "$MNT/etc/hostname"`. Update `/etc/hosts` `127.0.1.1` line: `sed -i -E 's/^127\.0\.1\.1\s+.*/127.0.1.1\t'"$SLUG"'/' "$MNT/etc/hosts"`. If no `127.0.1.1` line exists, append one.
    - **SSH host keys**: `rm -f "$MNT"/etc/ssh/ssh_host_*`. Do NOT regenerate at prep time — let sshd's first-boot service or first-boot.sh handle it on the appliance (cleaner; uses the appliance's own entropy). Add a marker file `"$MNT/etc/ssh/regenerate-on-boot"` so the operator can confirm via filesystem inspection that the prep ran.
    - **Tailscale identity**: `rm -f "$MNT/var/lib/tailscale/tailscaled.state" "$MNT/var/lib/tailscale/tailscaled.log"*`. Leave the directory and `tailscaled.sock` alone (sock recreates at runtime).
    - **Journals**: `rm -rf "$MNT/var/log/journal"/* 2>/dev/null || true`. Also clear `/var/log/syslog*`, `/var/log/auth.log*`, `/var/log/dmesg*` if they exist.
    - **Bash history**: `rm -f "$MNT/root/.bash_history"`. For each user dir under `$MNT/home/*`: `rm -f "$d/.bash_history" "$d/.lesshst" "$d/.viminfo" "$d/.python_history"`.
    - **Customer slug stamp**: `install -m 0644 /dev/stdin "$MNT/etc/mailbox-customer" <<<"$SLUG"`. Add a comment header line via `printf '# Set by factory-prep-nvme.sh on %s\n%s\n' "$(date -Iseconds)" "$SLUG" > "$MNT/etc/mailbox-customer"`.
    - **Permissions sanity**: `chmod 0644 "$MNT/etc/hostname" "$MNT/etc/hosts" "$MNT/etc/mailbox-customer"`; `chown 0:0` the same files.
12. `sync`. The `trap` handles umount.
13. Print summary: slug, device, what was wiped, what was set. Suggest next step: power-cycle on a spare Jetson to confirm clean boot, then pack.

**Exit codes:**
- 0 success
- 1 bad invocation / missing root
- 2 validation failure (bad slug, no ext4 partition, device not a block device)
- 3 safety gate tripped (root-device match, partition mounted on workstation, user declined)

`chmod +x` the file when done.
