## Overview — the two-phase pipeline

**The problem.** SDK Manager flash takes 45–90 minutes per unit and requires a host PC, a USB cable, and JetPack SDK Manager installed. At 3–5 customers per 90 days the per-unit SDK flash becomes the bottleneck — one install session per customer is not sustainable.

**The solution.** Flash exactly one Jetson once (the "seed unit"), run full factory bootstrap on it, capture the NVMe to a golden image, then `dd` that image onto blank NVMes at packing time. Each blank NVMe gets the full provisioned appliance state in ~15 minutes of unattended `dd` time, followed by a 2-minute personalization script. SDK Manager is never touched again until a JetPack version bump.

**Time savings.** Per-customer provisioning: hours (SDK Manager + model pulls + migrations) → ~15 min `dd` + 2 min `factory-prep-nvme.sh` + 5 min boot-verify.

```
Phase A — one-time per JetPack version
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ SDK Manager flash (STAQPRO-225)                                         │
  │  ↓                                                                      │
  │ factory-bootstrap.sh on seed Jetson (STAQPRO-410)                       │
  │  • Docker (JetsonHacks), nvidia-container-toolkit, GPU smoke             │
  │  • git clone, .env from template                                         │
  │  • migrations, Qdrant bootstrap                                          │
  │  • ollama pull qwen3:4b-instruct + nomic-embed-text:v1.5                 │
  │  • ollama create qwen3:4b-ctx4k (custom Modelfile, DR-18 / STAQPRO-330) │
  │  ↓                                                                      │

  │  ↓                                                                      │
```

## §2 — Capture the golden image (Phase A)

**Goal:** Produce a `mailbox-golden-v<JetPack>-<YYYY-MM-DD>.img.zst` on the workstation.

### 2.1 Identify the NVMe

Insert the seed NVMe into the workstation's M.2 NVMe slot or a USB-NVMe adapter. Wait ~5 seconds for the device to enumerate, then:

```bash
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
ls -l /dev/disk/by-id/ | grep nvme
```

Look for the seed NVMe in the `lsblk` output (it should be unmounted). Prefer the stable `/dev/disk/by-id/<id>` symlink over `/dev/nvmeXn1` — by-id names are stable across reboots and USB-adapter re-enumerations.

### 2.2 Dry-run first

Always preview before capturing. The script will print the planned action including source device, resolved path, size, output path, and available free space. If anything looks wrong, abort now.

```bash
sudo ./scripts/factory-image.sh --dry-run \
    --source /dev/disk/by-id/<id-of-seed-nvme> \
    --jetpack-version 6.2 \
    --output-dir /var/lib/mailbox-images
```

Expected dry-run output:
```
→ factory-image.sh — golden image capture plan
  Source (arg)       : /dev/disk/by-id/<id>
  Source (resolved)  : /dev/nvme1n1
  Source size        : 953.9G
  Output image       : /var/lib/mailbox-images/mailbox-golden-v6.2-2026-05-18.img
  Compressed output  : /var/lib/mailbox-images/mailbox-golden-v6.2-2026-05-18.img.zst
  Free space         : <available>
  Mode               : DRY RUN — no reads or writes
```

**Safety note:** The script refuses if `--source` resolves to the workstation's own root device. If you see:

```
REFUSED: --source resolves to the workstation's root device /dev/sda
```

**Stop immediately.** You have the wrong device path. Re-verify with `lsblk` and `findmnt /`.

### 2.3 Real capture

After confirming the dry-run looks correct, run the real capture. The script will prompt for `YES` before starting:

```bash
sudo ./scripts/factory-image.sh \
    --source /dev/disk/by-id/<id-of-seed-nvme> \
    --jetpack-version 6.2 \
    --output-dir /var/lib/mailbox-images
```

The capture reads the full NVMe (~954 GB for the Kingston SNV3S1000G). Using `pv` if available, otherwise `dd status=progress`. Expect ~15–20 minutes at USB-NVMe speeds; faster via M.2.

### 2.4 Expected output files

```
/var/lib/mailbox-images/
  mailbox-golden-v6.2-2026-05-18.img          ← raw image
  mailbox-golden-v6.2-2026-05-18.img.sha256   ← sha256 of raw image
  mailbox-golden-v6.2-2026-05-18.img.zst      ← zstd -19 compressed
  mailbox-golden-v6.2-2026-05-18.img.zst.sha256
```

The `.zst` compressed image is the long-term archive artifact. Verify it:

```bash
sha256sum --check /var/lib/mailbox-images/mailbox-golden-v6.2-2026-05-18.img.sha256
sha256sum --check /var/lib/mailbox-images/mailbox-golden-v6.2-2026-05-18.img.zst.sha256
```

## §3 — Per-customer flash (Phase B)

**Goal:** A blank NVMe with the golden image written and personalized for a specific customer.

### 3.1 Insert and identify the blank NVMe

Insert a blank (or don't-care-about-data) NVMe into the workstation M.2 slot or USB adapter. The seed NVMe must be removed first to avoid confusion.

```bash
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT
ls -l /dev/disk/by-id/ | grep nvme
```

Triple-check: the target device should be unmounted and NOT the workstation root disk.

### 3.2 Write the golden image (operator-typed `dd` — this is intentional)

**The `dd` step is deliberately operator-typed, not scripted.** Auto-`dd` is a foot-gun — forcing the operator to copy-paste the exact target by-id is the safety gate. This decision is intentional and permanent.

Decompress and write:

```bash
# Decompress the compressed image (skip if you have the raw .img)
zstd -d /var/lib/mailbox-images/mailbox-golden-v6.2-2026-05-18.img.zst \
     -o /tmp/golden.img

# SAFETY CHECK: confirm the target before writing
lsblk -o NAME,SIZE,MOUNTPOINT
#  → verify the target by-id is the blank NVMe, unmounted, NOT the workstation root

# Write the image — REPLACE <target-id> with the real by-id of the BLANK NVMe
sudo dd if=/tmp/golden.img \
        of=/dev/disk/by-id/<target-id> \
        bs=64M status=progress conv=fsync
sync

# Clean up decompressed temp image
rm /tmp/golden.img
```

Expect ~15 minutes at USB-NVMe speeds. `dd` will print throughput and total bytes when done.

### 3.3 Personalize with `factory-prep-nvme.sh`

After `dd` completes and `sync` returns, personalize the NVMe for this customer.

**Customer slug rules:** lowercase alphanumeric + hyphens, 2–32 chars, must not start or end with a hyphen. This is the same DNS-label regex used by `scripts/provision-customer-dns.sh` — intentionally so the slug is directly usable as `<slug>.mailbox.staqs.io` per NC-25.

```bash
# Dry-run first
sudo ./scripts/factory-prep-nvme.sh --dry-run \
    --slug <customer-slug> \
    --by-id /dev/disk/by-id/<target-id>

# Real personalization (operator types YES)
sudo ./scripts/factory-prep-nvme.sh \
    --slug <customer-slug> \
    --by-id /dev/disk/by-id/<target-id>
```

The script will:
1. Validate slug against the DNS-label regex — exits 2 on invalid slug before touching anything.
2. Refuse if the target is the workstation root disk (exits 3).
3. Mount the largest ext4 partition (the JetPack rootfs).
4. Wipe per-host identity (SSH keys, Tailscale state, journals, histories).
5. Stamp `/etc/hostname`, `/etc/hosts`, `/etc/mailbox-customer` with the slug.
6. Unmount cleanly via `trap EXIT` — even on Ctrl-C or OOM kill.

## §4 — Boot-verify on a spare Jetson

**Goal:** Confirm the personalized NVMe produces a clean, uniquely-identified appliance.

Insert the personalized NVMe into a spare bench Jetson Orin Nano Super (one that's NOT the seed unit). Power on. Connect via console or SSH when the device comes up.

- [ ] Hostname is the customer slug: `hostname` → should print `<customer-slug>`
- [ ] Customer stamp: `cat /etc/mailbox-customer` → should print `<customer-slug>`
- [ ] SSH host keys regenerated: `ls -la /etc/ssh/ssh_host_*` → files should exist and have a timestamp more recent than the factory-prep run (sshd generates them on first boot when they're absent)
- [ ] Regenerate marker cleaned: `ls /etc/ssh/regenerate-on-boot` → this file should NOT exist after a successful first-boot sshd run (sshd or a first-boot script removes it)
- [ ] Tailscale is a new node: `tailscale status` should show the device requesting enrollment with a new node key. Approve in the tailnet admin console; confirm it appears as a new device, not a duplicate of the seed.
- [ ] Docker Compose stack healthy: `docker compose ps` → services in Created or Up state (some will need manual start if `factory-bootstrap.sh` didn't configure an on-boot restart policy — verify during first bench run)
- [ ] Ollama models present: `docker exec mailbox-ollama-1 ollama list` → should list `qwen3:4b-ctx4k` and `nomic-embed-text:v1.5`

## §5 — Pack and ship

After boot-verify passes:

1. Power down the bench Jetson cleanly: `sudo shutdown -h now`. Wait for power LED to go dark.
2. Remove the personalized NVMe from the bench Jetson.
3. Install the NVMe into the **customer's** Jetson appliance.
4. Label the appliance with the customer slug and expected hostname: `<slug>.mailbox.staqs.io`.
5. Follow `docs/runbook/provisioning.v0.1.0.md` for the customer-LAN-side bring-up that follows (Tailscale enrollment, Caddy HTTPS, Gmail OAuth, smoke test).

The customer-DNS step (`scripts/provision-customer-dns.sh`) uses the same slug to create the `<slug>.mailbox.staqs.io` A record — the slug here and the slug there must match.

## §6 — Refreshing the golden image (when JetPack updates)

When NVIDIA releases a new JetPack version that MailBOX needs:

1. Repeat §1 with the new JetPack — flash a fresh seed Jetson, run `factory-bootstrap.sh`.
2. Repeat §2 to capture a new golden image. Use the new version string in `--jetpack-version` (e.g. `6.3`).
3. New filename: `mailbox-golden-v6.3-YYYY-MM-DD.img.zst` — old images are NOT deleted.

**Archive convention for old images:**

```bash
mkdir -p /var/lib/mailbox-images/archive
mv /var/lib/mailbox-images/mailbox-golden-v6.2-*.img.zst* \
   /var/lib/mailbox-images/archive/
```

Old golden images are kept for rollback — if a newly-shipped customer NVMe has issues, you can re-`dd` from the prior golden version while debugging. Don't delete until at least one successful customer using the new version has been deployed for 30+ days.

## §7 — Known limitations & follow-ups

- **No hardware verification yet.** Scripts have unit-shaped self-checks (root, slug regex, root-device refusal, `--dry-run` round-trip), but the full pipeline requires a real NVMe in the workstation NVMe slot. First end-to-end pass is Dustin at the bench and validates this runbook into v0.2.0.

- **`/etc/mailbox-customer` not yet wired into `.env` loader.** The slug stamp is a forensic/identification artifact — the dashboard's `.env` loader does not yet read it at boot. A separate dashboard change is needed to source `CUSTOMER_SLUG` from this file at container start. File a follow-up ticket. For now: after the appliance boots, set `CUSTOMER_SLUG=<slug>` in `/home/bob/mailbox/.env` manually as part of the provisioning runbook (§3 of `provisioning.v0.1.0.md`).

- **`factory-bootstrap.sh` (STAQPRO-410) lives outside this branch.** Do NOT modify it on `feat/staqpro-409-golden-image`. Reference by ticket only; coordinate with the STAQPRO-410 owner before making changes that affect seed-Jetson bootstrap behavior.

- **The `dd` step is deliberately operator-typed.** Auto-`dd` is a foot-gun. The manual copy-paste of the target by-id IS the safety gate. This will not be scripted.

- **Spare bench Jetson required for boot-verify (§4).** The seed Jetson is tied up as a reference unit and should not be used for boot-verify (would require re-flashing). A second Orin Nano Super bench unit is needed to complete the pipeline at scale. `OPEN Q:` Is there a bench unit available, or does every unit need to boot-verify in situ?

- **Image storage grows quickly.** A 954 GB NVMe produces a ~200–400 GB raw image (compressed depends on how much data is on the drive at capture time). Budget workstation disk space accordingly.
